# MER0028

## Summary

`MER0028` flags heavy multi-line expressions embedded directly in object and anonymous-object initializer members.

## Intent

Initializer members should read like value assembly, not like hidden business logic blocks. When the member expression itself spans many lines and carries substantial branching, operator density, or pipeline work, it becomes difficult to review the construction code.

## What It Reports

Reports:

```csharp
return new {
    RailPriority =
        row.TransportModeName == "rail"
        || row.TransportModeName == "train"
            ? 1
            : row.CommonName == "rail"
                || row.CommonName == "train"
                    ? 2
                    : row.CommonName == "station"
                        && row.TransportModeName != "bus"
                        && row.StopTypeName != "bus"
                        && row.TransportModeName != "tram"
                        && row.StopTypeName != "tram"
                            ? 3
                            : 99
};
```

```csharp
return new ScopePayload {
    AudienceProfiles = profiles
        .Select(profile => new {
            profile.Key,
            VisibleDomains = profile.VisibleDomains
                .Select(domain => domain.ToLowerInvariant())
                .OrderBy(domain => domain, StringComparer.Ordinal)
                .ToArray()
        })
        .OrderBy(profile => profile.Key, StringComparer.Ordinal)
        .ToArray()
};
```

```csharp
return new Payload {
    Priority =
        isPrimary
            ? 0
            : hasRailName
                ? 1
                : hasTrainName
                    ? 2
                    : looksLikeStation && !isBus && !isTram && !isMetro
                        ? 3
                        : 99
};
```

## What It Does Not Report

Does not report:

```csharp
var priority = ResolvePriority(row);

return new {
    RailPriority = priority
};
```

```csharp
return new OuterPayload {
    Quota = new InnerPayload {
        FeatureKey = "quota",
        Remaining = state.Remaining,
        IsDegraded = state.IsDegraded || state.IsMissingSnapshot || state.UsesFallbackLimit
    }
};
```

```csharp
return new Payload {
    Priority = isExactRail ? 0 : hasRailName ? 1 : 99
};
```

## Examples

Incorrect:

```csharp
return new CandidatePayload {
    Score =
        candidate.IsPrimary
            ? 0
            : candidate.HasNameHint
                ? 1
                : candidate.IsFallback && !candidate.IsSuppressed && !candidate.IsArchived && !candidate.IsHidden
                    ? 2
                    : 99
};
```

Correct:

```csharp
var score = ResolveCandidateScore(candidate);

return new CandidatePayload {
    Score = score
};
```

## Preferred Refactor

1. Compute the value before the initializer.
2. Move repeated logic into a named helper.
3. Keep initializer members focused on wiring already-understood values into the object shape.

## Edge Cases

- Generated code does not report.
- The rule applies to anonymous-object members and object initializer assignments.
- A nested object or anonymous-object construction used as the member value does not report by itself. The rule focuses on the direct member expression, not the entire nested construction subtree.
- Short inline expressions do not report even when they use a ternary.

## Suppression Guidance

Refactor first. Suppress only when keeping the full expression inline is materially clearer than staging the value before construction.
