# MER0006

## Summary

`MER0006` flags service resolution from `RequestServices` or `IServiceProvider` inside controller actions.

## Intent

The rule exists because service location in actions hides dependencies from signatures and weakens startup validation. Controller-wide dependencies belong in constructors; action-only dependencies belong in `[FromServices]` parameters.

## What It Reports

Reports:

```csharp
[HttpGet]
public ActionResult<ReportStatusDto> GetStatus() {
    var signer = HttpContext.RequestServices.GetRequiredService<IUrlSigningService>();
    return Ok();
}
```

```csharp
[HttpPost]
public ActionResult Queue() {
    var service = _serviceProvider.GetService<IReportQueueService>();
    return Accepted();
}
```

## What It Does Not Report

Does not report:

```csharp
[HttpGet]
public ActionResult<ReportStatusDto> GetStatus([FromServices] IUrlSigningService signer) {
    return Ok();
}
```

```csharp
public sealed class ReportCoordinator {
    public object Build(HttpContext httpContext) {
        return httpContext.RequestServices.GetRequiredService<IReportService>();
    }
}
```

The second example is outside the phase-one controller-action contract. Runtime services may be governed by a later rollout with allowlists.

## Examples

Incorrect:

```csharp
[HttpGet]
public Task<ReportDto> GetAsync() {
    var service = HttpContext.RequestServices.GetRequiredService<IReportService>();
    return service.GetAsync();
}
```

Correct:

```csharp
public ReportsController(IReportService service) {
    _service = service;
}

[HttpGet]
public Task<ReportDto> GetAsync() => _service.GetAsync();
```

### Action-Specific Dependency

Incorrect:

```csharp
[HttpPost]
public async Task<IActionResult> GenerateAsync() {
    var queue = HttpContext.RequestServices.GetRequiredService<IReportQueue>();
    await queue.EnqueueAsync();
    return Accepted();
}
```

Correct:

```csharp
[HttpPost]
public async Task<IActionResult> GenerateAsync([FromServices] IReportQueue queue) {
    await queue.EnqueueAsync();
    return Accepted();
}
```

### Captured Provider

Incorrect:

```csharp
public sealed class ReportsController : ControllerBase {
    private readonly IServiceProvider _services;

    public ReportsController(IServiceProvider services) {
        _services = services;
    }

    [HttpGet]
    public Task<ReportDto> GetAsync() {
        return _services.GetRequiredService<IReportReader>().GetAsync();
    }
}
```

Correct:

```csharp
public sealed class ReportsController : ControllerBase {
    private readonly IReportReader _reader;

    public ReportsController(IReportReader reader) {
        _reader = reader;
    }

    [HttpGet]
    public Task<ReportDto> GetAsync() {
        return _reader.GetAsync();
    }
}
```

## Current Rollout

- Live warning scope: `warning` in analyzer-scope backend projects during live analysis and explicit analyzer-enabled lanes
- Inventory command: `rtk pnpm backend:analyzers:inventory -- --diagnostics MER0006 --project apps/backend/Meridian.API/Meridian.API.csproj`
- Fallback loading check: `rtk pnpm backend:analyzers:validate:build -- --diagnostics MER0006`
- Candidate artifact: [Backend custom analyzer staged inventory rollout](docs/analysis/2026-05/2026-05-07-backend-custom-analyzer-staged-inventory.md)

`MER0006` is warning-enabled in committed `.editorconfig` for analyzer-scope live analysis and explicit analyzer-enabled lanes. Normal backend builds still keep analyzers off by default through `apps/backend/Directory.Build.props`. The rule intentionally covers controller actions first.

## Preferred Refactor

1. Move repeated dependencies to constructor injection.
2. Use `[FromServices]` for action-specific dependencies.
3. Keep framework edge cases in middleware, filters, handlers, hosted services, and composition roots out of this first rollout.

## Edge Cases

- The rule only analyzes HTTP action methods.
- The rule recognizes `GetService` and `GetRequiredService` calls on semantic `System.IServiceProvider` targets.
- The rule keeps a syntax fallback for unresolved analyzer shapes that use `RequestServices`, `ServiceProvider`, `serviceProvider`, `_serviceProvider`, or `provider` targets.
- Service registration lambdas and non-controller services are outside the current contract.

## Suppression Guidance

Default stance: use DI signatures instead of service location.

A suppression is only justified when ASP.NET requires a late-bound action dependency and `[FromServices]` cannot express it.
