# MER0017

## Summary

`MER0017` flags async materialization calls without an obvious `Where`, `Take`, or `Skip` bound.

## Intent

Unbounded EF materialization can load large tables or relationship graphs into memory. Runtime code should make its query bounds visible near the materialization call.

## What It Reports

Reports:

- `ToListAsync()`
- `ToArrayAsync()`
- `ToDictionaryAsync()`
- `ToHashSetAsync()`

The rule does not report when the same invocation chain visibly includes `Where`, `Take`, or `Skip`, or when the materialized local variable, local reassignment, or helper method return was initialized from a visible `Where`, `Take`, or `Skip` chain.

The rule also treats these as bounded for materialization-memory purposes:

- C# query expressions with a `where` clause.
- Aggregate materialization after `GroupBy`, because the result is the grouped projection rather than raw entity rows.
- Raw SQL materialization when the SQL argument visibly contains a bounding term such as `WHERE`, `LIMIT`, `OFFSET`, `TOP`, or a `p_limit` parameter.
- Explicit full-materialization boundaries whose method names declare complete-result semantics, catalogue/reference reads, report generation, or maintenance validation/repair. These are allowed because adding a hidden `Take` would break their contract.

## Examples

Incorrect:

```csharp
var reports = await dbContext.Reports.ToListAsync(cancellationToken);
```

Correct:

```csharp
var reports = await dbContext.Reports
    .Where(report => report.TenantId == tenantId)
    .Take(100)
    .ToListAsync(cancellationToken);
```

### Dictionary Materialization

Incorrect:

```csharp
var tenants = await dbContext.Tenants
    .ToDictionaryAsync(tenant => tenant.Id, cancellationToken);
```

Correct:

```csharp
var tenants = await dbContext.Tenants
    .Where(tenant => tenant.IsActive)
    .Take(500)
    .ToDictionaryAsync(tenant => tenant.Id, cancellationToken);
```

### Array Materialization

Incorrect:

```csharp
var jobs = await dbContext.ReportJobs.ToArrayAsync(cancellationToken);
```

Correct:

```csharp
var jobs = await dbContext.ReportJobs
    .Where(job => job.TenantId == tenantId)
    .OrderByDescending(job => job.CreatedAt)
    .Take(50)
    .ToArrayAsync(cancellationToken);
```

## Preferred Refactor

Add a visible filter/page/limit, or move deliberate full-table maintenance flows behind a named maintenance boundary and document why they are acceptable.

If the query is already bounded inside query syntax, a helper query method, aggregate SQL, or raw SQL text, prefer making that bound visible enough for the analyzer instead of adding duplicate runtime filters.

Do not satisfy this rule by adding arbitrary caps to methods whose public contract means "all", "full catalogue", or "validate the complete graph". For those methods, keep the contract accurate and make the full-materialization boundary explicit in the method name.
