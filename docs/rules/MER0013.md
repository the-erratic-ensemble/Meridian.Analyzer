# MER0013

## Summary

`MER0013` flags source-level backend layer-boundary violations.

## Intent

The rule exists because Meridian follows documented Clean Architecture dependency flow:

- `Meridian.Core` must not depend on `Meridian.API`, `Meridian.Infrastructure`, EF Core, or ASP.NET Core.
- `Meridian.Infrastructure` must not depend on `Meridian.API`.
- `Meridian.Shared` must not depend on backend projects.
- `Meridian.Analytics` must not use the standard `MeridianDbContext` / PostgreSQL repository boundary.

## What It Reports

Reports:

```csharp
// apps/backend/Meridian.Core/...
using Meridian.Infrastructure.Database;
```

```csharp
// apps/backend/Meridian.Core/...
using Microsoft.EntityFrameworkCore;
```

```csharp
// apps/backend/Meridian.Core/...
private Meridian.Infrastructure.Database.MeridianDbContext? _dbContext;
```

```csharp
// apps/backend/Meridian.Analytics/...
private readonly MeridianDbContext _dbContext;
```

## What It Does Not Report

Does not report:

```csharp
// apps/backend/Meridian.Core/...
using Meridian.Shared.Constants;
```

```csharp
// apps/backend/Meridian.Infrastructure/...
using Meridian.Core.Repositories;
```

## Examples

Incorrect:

```csharp
namespace Meridian.Core.Reports;

using Meridian.Infrastructure.Database;

public sealed class ReportPolicy {
    private readonly MeridianDbContext _dbContext;
}
```

Correct:

```csharp
namespace Meridian.Core.Reports;

public sealed class ReportPolicy {
    private readonly IReportPolicyReader _reader;
}
```

### Infrastructure Depending on API

Incorrect:

```csharp
namespace Meridian.Infrastructure.Reports;

using Meridian.API.Features.Reports.Models;

public sealed class ReportRepository {
    public Task<ReportDto> ReadAsync(Guid id) {
    }
}
```

Correct:

```csharp
namespace Meridian.Infrastructure.Reports;

using Meridian.Core.Reports;

public sealed class ReportRepository {
    public Task<ReportRecord> ReadAsync(Guid id) {
    }
}
```

### Analytics Using API DbContext

Incorrect:

```csharp
namespace Meridian.Analytics.Services;

using Meridian.Infrastructure.Database;

public sealed class AnalyticsQueryService {
    private readonly MeridianDbContext _dbContext;
}
```

Correct:

```csharp
namespace Meridian.Analytics.Services;

public sealed class AnalyticsQueryService {
    private readonly IClickHouseAnalyticsStore _store;
}
```

## Current Rollout

- Live warning scope: none
- Inventory command: run per project, for example `rtk pnpm backend:analyzers:inventory -- --diagnostics MER0013 --project apps/backend/Meridian.Core/Meridian.Core.csproj`
- Fallback loading check: `rtk pnpm backend:analyzers:validate:build -- --diagnostics MER0013`
- Candidate artifact: [Backend custom analyzer staged inventory rollout](docs/analysis/2026-05/2026-05-07-backend-custom-analyzer-staged-inventory.md)

`MER0013` launches inventory-only. It catches source-level namespace/type drift; project-reference and package-graph rules still belong in MSBuild validation, CI scripts, or architecture tests.

## Preferred Refactor

1. Move contracts inward and implementations outward.
2. Replace forbidden references with Core/Shared abstractions where a boundary crossing is real.
3. Keep Analytics on its ClickHouse-specific storage boundary instead of adding `MeridianDbContext` usage.

## Edge Cases

- The rule currently analyzes file paths to identify backend projects.
- Fully qualified forbidden namespace references are reported even when no `using` directive exists.
- `Meridian.Analytics` reports `using Meridian.Infrastructure.Database` and direct `MeridianDbContext` identifiers.
- Fully dynamic project/package references are outside the Roslyn source rule.

## Suppression Guidance

Default stance: move the dependency to the correct layer.

A suppression is only justified when an active ADR explicitly permits the boundary crossing and the code links to that decision.
